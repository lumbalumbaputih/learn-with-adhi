The big picture

10 min

Before we dive in, you need a mental map of how everything connects. Let's use an analogy you already understand.

Think of a restaurant.
Your website or app is a restaurant. There are distinct parts that work together, and none of them is "the code." The code is just the recipe. Here's the full picture:

Every module in this guide covers one of those boxes. By the end, you'll know what each one does, how they connect, and how to work with them, even though AI writes the actual code.

Key takeaway

Coding is just one skill in a chain of ten. Publishing, connecting services, managing versions, and understanding file structures are equally important. These are the skills this guide teaches you.

2

The three files behind every website

15 min

Every website you've ever visited is made of just three types of files. You don't need to write them (AI does that), but you need to recognise what they are, like knowing the difference between flour, eggs, and sugar even if a chef does the baking.

Think of a human body.
HTML is the skeleton: the structure, the bones. It defines what's on the page (headings, paragraphs, buttons, images).

CSS is the clothing and appearance: colours, fonts, spacing, layout. It makes the skeleton look good.

JavaScript (JS) is the muscles and brain: it makes things move and react. Click a button? JS handles that. Load new data without refreshing? JS.

How they work together

When AI gives you code, it usually comes in one of these forms:

  • A single .html file with CSS and JS embedded inside it (simplest, most common)
  • Separate files: index.html + style.css + app.js (cleaner for bigger projects)
  • A React/Vue project (we'll cover this in Module 5)

What you actually need to know

When AI gives you an HTML file, just save it with the .html extension and open it in your browser. It works immediately. No installation needed. That's your first website right there.

Folder structure matters

Every project has a folder structure. Think of it like a filing cabinet: things need to be in the right drawer. Here's what a typical project looks like:

my-project/ ├── index.html # Main page (the "front door") ├── style.css # How it looks ├── app.js # What it does ├── images/ # Folder for pictures │ ├── logo.png │ └── hero.jpg └── README.md # Description of the project
Important: The file named index.html is special. Hosting services automatically look for this file as the homepage. Always name your main page index.html.
3

The terminal: texting your computer

15 min
Think of it as WhatsApp, but for your computer.
Instead of clicking buttons and menus, you type short commands. The computer reads your message and does what you asked. That's it. It looks scary because of the black screen, but it's literally just texting.

How to open it

Mac: Press Cmd + Space, type "Terminal", press Enter

Windows: Press Win key, type "PowerShell" or "Command Prompt", press Enter

Alternative (both): Install VS Code (free code editor) and press Ctrl + ` to open the built-in terminal

The only commands you need

You don't need to memorise hundreds of commands. Here are the 8 that cover 95% of what you'll do:

cd folder-name
Go into a folder. Like opening a drawer.
cd ..
Go back one folder. Close the drawer.
ls
List what's in the current folder. (Windows: dir)
mkdir name
Create a new folder.
touch file.html
Create a new empty file. (Windows: New-Item file.html)
rm file.html
Delete a file. Be careful, no trash bin!
clear
Clear the screen. Fresh start.
code .
Open current folder in VS Code.

Real example: setting up a project

$ mkdir my-website # Create a folder $ cd my-website # Go inside it $ touch index.html # Create main file $ touch style.css # Create style file $ code . # Open in VS Code

That's it. Five commands and you have a project ready. Now paste the AI-generated code into those files.

Pro tip

You don't need to type full folder names. Type the first few letters and press Tab. The terminal will auto-complete it for you. This is called "tab completion" and it saves enormous amounts of time.

4

Git and GitHub: save points for your project

20 min
Think of a video game.
When you play a game, you save your progress. If you mess up, you load the last save. Git does the same thing for your project files. Every time you "commit," you create a save point. You can always go back.

GitHub is the cloud. It's where your save files are backed up online. It also lets other people (or deployment services) access your code.

Git vs GitHub: they're different

Git GitHub
Software on your computer Website (github.com)
Tracks changes locally Stores your code online
Free, built into most systems Free for public projects
The engine The garage where you park

The 4 Git commands you'll actually use

# FIRST TIME: connect your folder to Git $ git init # Start tracking this folder # EVERY TIME you want to save progress: $ git add . # Stage all changes (put items in the box) $ git commit -m "added homepage" # Save with a note (seal the box) $ git push # Upload to GitHub (ship the box)
The postal service analogy:
git add = putting items in a box
git commit = sealing the box and writing a label on it
git push = sending the box to the post office (GitHub)

You can put items in the box many times, but nothing ships until you push.

Setting up GitHub (one-time)

Go to github.com and create a free account

Click the green "New" button to create a repository (this is your online project folder)

Follow the instructions GitHub gives you to connect your local folder. It's usually two commands you copy and paste.

From now on, every git push uploads your latest code to this repository.

What's a repository?

A repository (or "repo") is just a project folder that Git is tracking. When people say "check out my repo," they mean "look at my project on GitHub." Nothing fancy.

5

Publishing your site to the world

15 min

You have your files. They work on your computer. Now you want the world to see them. This process is called deployment (or "deploying").

Think of uploading a video to YouTube.
You have a video file on your computer. YouTube gives you a URL where anyone can watch it. Deploying a website works the same way. You upload files, get a URL, done.

Three free options (ranked by ease)

Platform Difficulty Best for How it works
Netlify Easiest Simple sites, portfolios Drag and drop your folder
Vercel Easy React/Next.js apps Connect your GitHub repo
GitHub Pages Free Simple HTML sites Push to a special branch

Deploying with Netlify (the easiest path)

Go to app.netlify.com/drop

Drag your entire project folder onto the page

Wait 10 seconds. Netlify gives you a URL like random-name-123.netlify.app

Your site is now live. Share the URL with anyone.

Important: Make sure your project has an index.html file at the top level. Netlify (and every hosting service) looks for this file first. If it's buried inside a subfolder, the site won't load properly.

The deploy flow

You have files on your computer
↓
upload / push
Hosting service builds and serves them
↓
assigns
You get a public URL
↓
anyone visits
The world sees your site

Auto-deploy from GitHub

Here's where it gets powerful. Both Netlify and Vercel can connect to your GitHub repo. After that, every time you git push, your site automatically updates. No manual uploading. You push code, the site refreshes in about 30 seconds.

6

APIs: how apps talk to each other

20 min

This is the big one. Almost every useful app connects to some external service: weather data, maps, payments, AI. All of these connections happen through APIs.

Think of a waiter at a restaurant.
You (the app) sit at a table. The kitchen (the service) has the food (data). You can't walk into the kitchen. Instead, you tell the waiter (the API) what you want. The waiter goes to the kitchen, gets your order, and brings it back to you in a standard format.

The menu is the API documentation: it tells you what you can order.
Your order is the API request.
The food is the API response.

What does an API response look like?

APIs send data in a format called JSON (JavaScript Object Notation). It looks like this:

{ "city": "Perth", "temperature": 24, "condition": "Sunny", "humidity": 45 }

It's just labelled data. "city" is the label, "Perth" is the value. Your app reads this and displays it however it wants.

API keys: your membership card

Most APIs require an API key: a unique string of characters that identifies you. It's like a membership card. Without it, the kitchen won't serve you.

# Example API key (never share the real one!) API_KEY=sk-abc123def456ghi789
Never put your API key directly in your code. If you push that code to GitHub, anyone can see and steal it. We'll cover how to hide it properly in Module 8 (Environment Variables).

Common APIs you'll use

API What it does Free tier?
OpenAI / Anthropic AI text generation Pay per use
Google Maps Maps, directions Limited free
Stripe Accept payments Free (pay per transaction)
OpenWeather Weather data Yes
Supabase Database + auth Yes (generous)
Firebase Database + hosting Yes
7

Domains and DNS: the phone book of the internet

10 min
Think of a phone book.
Every website lives on a server that has a number-based address (like a phone number): 172.67.182.12. Nobody can remember that. A domain name like google.com is just a human-friendly label that maps to that number.

DNS (Domain Name System) is the phone book. When you type a URL, DNS looks up the number and connects you.

Getting your own domain

Go to a domain registrar: Namecheap, Cloudflare, or Google Domains

Search for the name you want (e.g., adhiproject.com)

Buy it. Typical cost: $10-15/year for .com

In your hosting service (Netlify/Vercel), go to domain settings and add your custom domain. Follow their instructions to update DNS records.

You don't need a custom domain to start

Every hosting platform gives you a free subdomain like your-project.netlify.app or your-project.vercel.app. This is perfectly fine for learning, portfolios, and personal projects. Buy a custom domain only when you're ready to look professional.

What are DNS records?

When you connect a custom domain, you'll see terms like "A record" and "CNAME." Don't panic. Here's the simple version:

Record type What it does Plain English
A record Points domain to an IP address "This name lives at this number"
CNAME Points domain to another domain "This name is the same as that name"

Your hosting service always tells you exactly which records to add. You just copy and paste.

8

Environment variables: your secret vault

10 min
Think of a locked safe in your house.
Your API keys, passwords, and secret values live in this safe. Your code knows how to open the safe and grab what it needs, but the safe itself is never shown to anyone. If someone looks at your code on GitHub, they see the instruction "open the safe," not the contents inside.

How it works: the .env file

You create a file called .env in your project folder. It looks like this:

# .env file (NEVER upload this to GitHub) OPENAI_API_KEY=sk-abc123def456ghi789 DATABASE_URL=postgres://user:pass@host/db SECRET_KEY=my-super-secret-value

Your code reads from this file using something like process.env.OPENAI_API_KEY. But the .env file itself stays on your computer.

Keeping secrets safe: the .gitignore file

A .gitignore file tells Git which files to never upload. You always put .env in it:

# .gitignore .env node_modules/

Now when you git push, your secrets stay local and your code goes to GitHub without them.

What about on the hosting platform?

When your code runs on Netlify or Vercel, it needs those secrets too. But you don't upload the .env file. Instead, you type them directly into the hosting dashboard:

Go to your project settings on Netlify/Vercel

Find "Environment Variables" section

Add each key and value manually (e.g., key: OPENAI_API_KEY, value: sk-abc123...)

Save and redeploy. Your live site now has access to the secrets.

Golden rule: If a value starts with sk-, api_, secret_, or looks like a password, it belongs in a .env file, never in your code directly.

Package managers: the app store for code

20 min

In Level 1, you learned that AI writes code for you. But here's the thing: AI doesn't write everything from scratch. It uses pre-built pieces made by other developers. These pieces are called packages (or libraries, or dependencies).

Think of IKEA furniture.
You don't chop down a tree to make a desk. You go to IKEA, grab a flatpack, and assemble it. Packages work the same way. Someone already built "a tool for handling dates" or "a tool for making charts." Your project just grabs those tools and uses them.

A package manager is the IKEA store itself. You tell it what you need, it downloads it for you.

The two package managers you'll encounter

npm pip
For JavaScript / web projects For Python projects
Comes with Node.js Comes with Python
npm install package-name pip install package-name
Stores info in package.json Stores info in requirements.txt

How it works in practice

When AI gives you a project, it often says "run npm install" or "run pip install -r requirements.txt." Here's what actually happens:

AI gives you code + a list of packages it needs
↓
you run
npm install (or pip install)
↓
package manager
Downloads all packages from the internet
↓
stores them in
node_modules/ folder (npm) or site-packages/ (pip)

The key files you'll see

package.json

This is the recipe card for a JavaScript project. It lists the project's name, version, and most importantly, all the packages it depends on. When you run npm install, npm reads this file and downloads everything listed.

// package.json (simplified) { "name": "my-weather-app", "version": "1.0.0", "dependencies": { "express": "^4.18.0", // web server package "axios": "^1.6.0", // makes API calls "dotenv": "^16.3.0" // reads .env files } }
Never upload the node_modules folder to GitHub. It can contain thousands of files and be hundreds of megabytes. It's already in your .gitignore from Level 1. When someone else clones your project, they just run npm install and it rebuilds from package.json.

Commands you'll actually use

npm install
Download all packages listed in package.json
npm install axios
Add a specific new package to your project
npm run dev
Start your project in development mode
npm run build
Package your project for deployment
2

Node.js: JavaScript outside the browser

15 min

In Level 1 you learned JavaScript makes websites interactive. But JavaScript used to be trapped inside the browser. Then Node.js was created, and it changed everything.

Think of a fish that learned to breathe on land.
JavaScript was a fish that only lived in the browser (water). Node.js gave it lungs. Now JavaScript can run on servers, on your computer, build apps, talk to databases, and do basically everything, not just make buttons clickable on a webpage.

Why should you care?

When AI builds you a project that's more than a simple HTML page, it almost always uses Node.js. Every time you see these commands, Node.js is running behind the scenes:

$ npm install # npm is part of Node.js $ npm run dev # starts a local dev server via Node.js $ npx create-next-app # scaffolds a project using Node.js

Installing Node.js

Go to nodejs.org

Download the LTS version (Long Term Support = the stable one)

Install it like any normal app

Open your terminal and type node --version. If you see a number (like v20.11.0), it worked.

What gets installed with Node.js

When you install Node.js, you actually get three things: node (the JavaScript engine), npm (the package manager from Module 1), and npx (a tool to run packages without installing them). That's why npm "just works" after installing Node.js.

Frontend vs Backend

This is a good time to understand a distinction you'll hear constantly:

Frontend is everything the user sees: the webpage, the buttons, the layout. It runs in the browser.

Backend is the hidden engine: it processes data, talks to databases, handles login systems, and connects to APIs. It runs on a server using Node.js or Python.

When AI builds you a "full-stack" project, it's building both sides.

Frameworks: pre-built kitchens

20 min
Think of building a house.
You could lay every brick by hand (plain HTML/CSS/JS). Or you could buy a prefab house that already has walls, plumbing, and wiring, and then just customise the interior. A framework is that prefab house. It gives you structure so you can focus on the unique parts.

When AI builds anything more complex than a single-page site, it uses a framework. You need to recognise the big ones.

Frontend frameworks (what the user sees)

Framework Made by When AI uses it You'll recognise it by
React Meta Most interactive apps, dashboards .jsx files, useState, import React
Next.js Vercel Full websites with pages and routing pages/ or app/ folder, next.config.js
Vue Community Simpler interactive apps .vue files, <template> blocks

Backend frameworks (the engine room)

Framework Language When AI uses it You'll recognise it by
Express JavaScript Simple APIs, web servers app.get(), app.post(), app.listen()
FastAPI Python Data-heavy APIs, ML projects @app.get, uvicorn, .py files
Django Python Full web apps with admin panels manage.py, models.py, views.py

How to start a framework project

AI usually tells you to run a "create" command. These scaffold (build the skeleton of) a full project for you:

# Create a React app $ npx create-react-app my-app # Create a Next.js app $ npx create-next-app my-app # Create a Vue app $ npm create vue@latest my-app

After running one of these, you get a full project folder with everything pre-configured. Then you cd my-app, run npm install, and npm run dev to see it in your browser.

Which framework should you ask AI to use?

If you're not sure, ask AI to use Next.js. It handles both frontend and backend, deploys easily on Vercel, and is the most common choice in 2025/2026. For simple one-page tools, plain HTML is still fine.

4

Databases: where your data lives

20 min

Every app that saves anything (user accounts, posts, orders, scores) needs a database. You won't build one yourself, but you need to understand what they are so you can tell AI what you need.

Think of a spreadsheet vs a filing cabinet.
A SQL database (like PostgreSQL) is a spreadsheet: rows, columns, everything neatly organised. You know exactly where each piece of data lives.

A NoSQL database (like MongoDB or Firebase) is a filing cabinet with folders. Each folder can contain different things. More flexible, less structured.

SQL vs NoSQL at a glance

SQL NoSQL
Structure Tables with rows and columns Collections of documents (like JSON)
Best for Structured data (users, orders, products) Flexible data (chat messages, logs)
Popular ones PostgreSQL, MySQL, SQLite MongoDB, Firebase, DynamoDB
Query language SQL (its own language) Varies by platform

The easy route: Database-as-a-Service

You don't need to install or manage a database yourself. These platforms give you a database with a dashboard, and AI can connect to them with just a URL:

Platform Type Best for Free tier
Supabase SQL (PostgreSQL) Apps with user accounts, structured data Generous
Firebase NoSQL Real-time apps, chat, mobile Generous
PlanetScale SQL (MySQL) Large-scale web apps Limited
MongoDB Atlas NoSQL Flexible document storage Yes (512MB)

How the connection works

You create a database on Supabase/Firebase
↓
platform gives you
A connection URL (looks like a long string)
↓
you put it in
Your .env file as DATABASE_URL=...
↓
your code
Reads the URL and connects to the database
# In your .env file DATABASE_URL=postgresql://user:[email protected]:5432/postgres

What to tell AI

When you need a database, tell AI: "I need to store [what kind of data]. Use Supabase (or Firebase). Here's my connection URL: [from your .env]." That's enough for AI to set up the entire database connection for you.

Browser DevTools: your X-ray glasses

15 min
Think of X-ray glasses for websites.
Every browser has a hidden panel that lets you see the skeleton (HTML), the clothing (CSS), and the nervous system (JS) of any website. You can even change things live and see what happens. It's the single most useful debugging tool you have.

How to open DevTools

Any browser: Right-click anywhere on a page and click "Inspect"

Keyboard shortcut: Press F12 or Cmd+Option+I (Mac) / Ctrl+Shift+I (Windows)

The 4 tabs you need to know

Elements
See and edit the HTML and CSS live. Hover over elements to highlight them on the page. Great for "why is this button in the wrong place?"
Console
Shows error messages and log outputs. If something is broken, the error is almost always here. This is the tab you'll use most.
Network
Shows every request your page makes (API calls, images loading, etc). If an API call fails, you'll see it here in red.
Application
Shows stored data: cookies, local storage, session data. Useful for debugging login issues.

The Console is your best friend

When something breaks, open the Console tab first. It shows red error messages telling you what went wrong and on which line. You can copy that error message and paste it directly to AI with "fix this error" and AI will know what to do.

Real example: debugging a broken API call

// You open Console and see this: GET https://api.weather.com/data?key=undefined 401 (Unauthorized) // This tells you: // 1. It's trying to call the weather API // 2. key=undefined means the API key is missing // 3. 401 means "you're not authorized" // 4. Fix: check your .env file, the API key isn't loading

Reading error messages: they're trying to help you

20 min

Error messages look terrifying. A wall of red text feels like the computer is yelling at you. But here's the secret: error messages are the most helpful thing in programming. They tell you exactly what went wrong.

Think of a doctor's diagnosis.
"You have a broken bone in your left wrist" is scary to hear, but it tells you exactly what's wrong and where. An error message does the same thing. You just need to know how to read it.

Anatomy of an error message

TypeError: Cannot read properties of undefined (reading 'map') at ProductList (src/components/ProductList.jsx:14:22) at renderWithHooks (react-dom.development.js:15486:18)

Let's break that down into three parts:

Error type: TypeError. This tells you the category. TypeError means you tried to use something that doesn't exist or has the wrong type.

Message: Cannot read properties of undefined (reading 'map'). This is the actual problem. It tried to use .map() on something that was undefined (empty/missing).

Location: src/components/ProductList.jsx:14:22. The file, line number, and character position. Line 14 is where the problem is.

Common errors and what they actually mean

Error Plain English Likely fix
404 Not Found The page or API URL doesn't exist Check the URL for typos
401 Unauthorized Your API key is missing or wrong Check your .env file
500 Internal Server Error The server crashed Check your backend code/logs
CORS error The server blocks your browser from calling it Tell AI "I'm getting a CORS error"
Module not found A package is missing Run npm install
SyntaxError Typo in the code (missing bracket, comma) Check the line number
EADDRINUSE The port is already taken by another app Close the other app or change the port

The debugging workflow

When something breaks:
1. Read the error message (at least the first two lines)
2. Copy the error
3. Paste it to AI and say "I got this error, here's my code: [paste code]"
4. AI fixes it 90% of the time

The skill isn't fixing the error. The skill is finding and reading the error correctly so AI can fix it.

Reading documentation: the instruction manual

15 min

Every API, framework, and service has documentation (often called "docs"). This is the instruction manual. You won't read all of it, but knowing how to navigate it is a real superpower.

Think of a car manual.
You don't read it cover to cover. But when your dashboard shows a warning light, you flip to that section. Documentation works the same way. You search for the specific thing you need.

What to look for in docs

Quick Start
The minimum steps to get something working. Always start here.
API Reference
Lists every function/endpoint available and what data to send.
Examples
Working code samples you can copy and modify.
Authentication
How to set up your API key and get access.

How to use docs + AI together

The most effective workflow is:

Find the docs page for the service you want to use (Google "[service name] docs")

Skim the Quick Start to understand the basics

Copy the URL of the relevant docs page

Paste it to AI: "Use this API according to these docs: [URL]. Build me [what you want]."

This gives AI the exact specifications it needs to build things correctly. Without docs, AI sometimes guesses wrong about how an API works.

Example: popular documentation sites

Service Docs URL
Supabase supabase.com/docs
Next.js nextjs.org/docs
Vercel vercel.com/docs
Anthropic (Claude API) docs.anthropic.com
Stripe (payments) stripe.com/docs
OpenAI platform.openai.com/docs

You don't need to understand everything

Most docs pages have code examples in multiple languages. You don't need to understand the code. You just need to find the right section and hand it to AI. Think of yourself as the project manager: you find the requirements, AI does the implementation.

Docker: shipping your entire kitchen

20 min

Here's a problem you'll hit eventually: your app works perfectly on your computer, but when you deploy it or someone else tries to run it, everything breaks. Different versions, missing tools, wrong settings.

Think of a shipping container.
Before shipping containers, moving goods was chaos: different trucks, different loading systems, things breaking in transit. Then someone invented a standard metal box. Put anything inside, and it ships identically to any port in the world.

Docker is that shipping container for software. You pack your app, all its dependencies, the correct Node.js version, everything, into one container. That container runs identically on your laptop, your colleague's laptop, and the production server.

The key Docker concepts

Term What it is Analogy
Image A blueprint/snapshot of your app and its environment The IKEA instruction manual + all parts in a sealed box
Container A running instance of an image The assembled furniture in your room
Dockerfile A text file with instructions to build an image The step-by-step assembly guide
Docker Hub An online library of pre-built images The IKEA catalogue

What a Dockerfile looks like

When do you need Docker?

For simple HTML sites and Netlify/Vercel deployments, you don't. Docker becomes important when: you're deploying to platforms like AWS or Google Cloud, your app uses multiple services (database + backend + frontend), or you need your app to run identically everywhere. If AI generates a Dockerfile for your project, now you know what it does.

Authentication: who's at the door?

20 min

If your app has user accounts (login, signup, profiles), you need authentication (who you are) and authorisation (what you're allowed to do).

Think of a hotel.
Authentication = checking your ID at the front desk. "Are you who you say you are?"
Authorisation = your room key only opens your room, not the presidential suite. "What are you allowed to access?"

The approaches, ranked by complexity

Method How it works When to use
Third-party auth (easiest) Use Supabase Auth, Firebase Auth, or Clerk. They handle everything. Almost always. Start here.
OAuth / "Sign in with Google" User logs in with their Google/GitHub/Apple account. You never see their password. When you want social login
JWT tokens User logs in, gets a token (like a digital wristband). Token proves identity on every request. Custom APIs, advanced use
Roll your own Build login/signup from scratch, hash passwords, manage sessions. Almost never. Serious security risk.
Never build authentication from scratch. Password hashing, session management, and security vulnerabilities are extremely easy to get wrong. Always use a third-party service. This is one area where even experienced developers use pre-built tools.

Serverless functions: backend without a server

20 min
Think of a vending machine vs a restaurant.
A restaurant (traditional server) is always open, always staffed, always costing money, even when nobody is eating. A vending machine (serverless) sits idle until someone puts in a coin. It only runs (and costs money) when someone actually uses it.

Serverless functions are tiny pieces of backend code that only run when triggered. No server to manage, no monthly cost when idle.

Why serverless matters for you

Remember in Level 1, we said you shouldn't put API keys in frontend code? Serverless functions solve this. Your frontend calls your own serverless function, and the function (running safely on the server) calls the external API with the secret key.

User clicks "Get weather"
↓
frontend calls
Your serverless function (/api/weather)
↓
function uses secret API key to call
External weather API
↓
returns data
User sees the weather

The API key never touches the browser. It lives safely in the serverless function's environment variables.

Where to run serverless functions

Platform How to create Free tier
Vercel Create files in /api/ folder 100,000 calls/month
Netlify Functions Create files in /netlify/functions/ 125,000 calls/month
Supabase Edge Functions Deploy via Supabase CLI 500,000 calls/month

Example: a Vercel serverless function

// File: /api/weather.js // This file IS the serverless function. // Vercel automatically turns it into an API endpoint. export default async function handler(req, res) { // This code runs on the server, not in the browser const apiKey = process.env.WEATHER_API_KEY; // safe! const data = await fetch(`https://api.weather.com?key=${apiKey}`); const json = await data.json(); res.json(json); // send data back to frontend }

When to use serverless

Use serverless functions when: you need to hide API keys from the frontend, you need to process data before sending it to the user, you want to send emails or notifications, or you need a lightweight backend without managing a full server. For most projects that AI builds for you, serverless is the right choice.

Webhooks: "call me when it happens"

15 min
Think of a doorbell.
You don't stand at your door all day checking if someone's there (that would be exhausting). Instead, you install a doorbell. When someone arrives, the doorbell notifies you.

A webhook is a doorbell for your app. Instead of constantly checking "did a payment come in? did a form get submitted?", you give the external service a URL. When the event happens, the service calls your URL automatically.

How webhooks work

You tell Stripe: "when a payment succeeds, call this URL"
↓
days later, a customer pays
Stripe sends data to your URL automatically
↓
your serverless function
Receives the data and acts on it

Common webhook uses

Service Event Your app does
Stripe Payment succeeds Activate subscription, send receipt
GitHub Code pushed Trigger deployment (this is how CI/CD works!)
Typeform Form submitted Save response to database, send email
Shopify Order placed Update inventory, notify warehouse

Webhooks + Serverless = perfect pair

A webhook needs a URL to call. A serverless function gives you a URL that runs code. Together, they let you react to events from any service without running a permanent server. Tell AI: "Create a serverless function that receives a webhook from [service] when [event happens] and then [what you want to do]."

Security basics: locking the doors

15 min

You don't need to be a security expert, but you need to know the basics so you don't accidentally leave the front door wide open.

The security checklist

HTTPS everywhere
Always use https:// not http://. Netlify and Vercel do this automatically. HTTPS encrypts data in transit.
Never expose secrets
API keys in .env files, never in code. Check your GitHub repo: can you see any keys? If yes, rotate them immediately.
Use third-party auth
Never store plain-text passwords. Use Supabase Auth, Firebase Auth, or Clerk. They handle hashing and security.
Validate user input
Never trust data from users. If you expect a number, check it's a number. Tell AI "validate all user inputs."
Keep packages updated
Run npm audit to check for known vulnerabilities. Run npm audit fix to patch them.
CORS settings
Control which websites can call your API. Tell AI "set up CORS to only allow requests from my domain."
If you accidentally push an API key to GitHub: assume it's compromised. Go to the service's dashboard, revoke/regenerate the key immediately, and replace it in your .env file. GitHub bots scan for leaked keys within minutes.

Planning projects: the architect's blueprint

25 min

This is the most important module. Everything you've learned across all three levels comes together here. Your job isn't to code. Your job is to be the architect: design the plan, then hand it to AI to build.

Think of building a house.
The architect doesn't lay bricks. They draw blueprints, choose materials, plan the plumbing, and decide where the doors go. Then the builder (AI) follows the blueprint. A good blueprint = a good house. A vague "just build me something" = a mess.

The project brief template

Before asking AI to build anything, write a brief. Here's the template:

## PROJECT BRIEF What it does: [One sentence. "A weather dashboard that shows forecasts for Australian cities."] Who uses it: [Target user. "Me, personally" or "Students in my course" or "Customers buying fish"] Core features (ranked by priority): 1. [Must have] Search for a city and see the 5-day forecast 2. [Must have] Save favourite cities 3. [Nice to have] Dark mode 4. [Nice to have] Weather alerts Tech choices: - Frontend: Next.js (deploy on Vercel) - Database: Supabase (for saved cities) - API: OpenWeather API - Auth: Supabase Auth (if user accounts needed) What I already have: - OpenWeather API key - Supabase project URL - Vercel account

The build sequence

Don't ask AI to build everything at once. Break it into phases:

Phase 1: Skeleton. "Set up a Next.js project with a homepage that has a search bar. No API calls yet, just the layout." Deploy this first to make sure hosting works.

Phase 2: Core feature. "Connect the search bar to the OpenWeather API. Show temperature and condition for the searched city." Test this thoroughly.

Phase 3: Data persistence. "Connect to Supabase. Let users save favourite cities. Store them in a 'favourites' table." Test saving and loading.

Phase 4: Auth (if needed). "Add Supabase Auth so each user has their own saved cities." Test sign up, login, logout.

Phase 5: Polish. "Add dark mode, improve the design, add loading states and error handling."

Why phases matter

If you ask AI to build everything at once and something breaks, you won't know what caused it. Building in phases means each step works before you add the next. It's like building a house: foundation first, then walls, then roof, then paint. Never paint before the roof is on.

How to communicate with AI effectively

Bad prompt Good prompt
"Build me a weather app" "Build a Next.js weather app using OpenWeather API. Show 5-day forecast. Use Tailwind CSS for styling. Deploy on Vercel."
"It's broken, fix it" "I'm getting this error [paste error]. Here's the file it mentions [paste code]. I think it's related to the API call."
"Add a database" "Add Supabase. Create a 'cities' table with columns: id, user_id, city_name, created_at. Here's my Supabase URL from .env."
"Make it look better" "Use a clean design with rounded cards, a blue colour scheme, and responsive layout for mobile. Reference this site for style: [URL]."

The golden rule of working with AI

Be specific about what you want and which tools to use. Let AI decide how to implement it. You are the architect. AI is the builder. The more detailed your blueprint, the better the result.
